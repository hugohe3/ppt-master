# 什么是 PPT？

[English](../what-is-ppt.md) | [中文](./what-is-ppt.md)

---

本文说明人们所说的“PPT”究竟指什么、人们为什么制作演示文稿、PowerPoint 文档如何构成，以及这种媒介适合或不适合哪些沟通任务。

它把 PPT 同时视为沟通媒介、可编辑文档、传递载体和原生文件结构，而不只是若干页面。

## 1. “PPT”同时指代几种不同对象

日常语境中的“PPT”是一个混合概念。讨论时至少需要区分五种含义：

| 含义 | 定义 |
|---|---|
| PowerPoint | Microsoft 提供的创作、编辑、演示与协作软件 |
| Presentation | 一次包含信息、受众、传递场景和预期结果的沟通活动 |
| Deck | 在活动前、中、后使用的有序、可编辑演示文档 |
| Slide show | 被现场演示或自动播放的页面序列，可包含转场、动画、旁白、视频和计时 |
| PPT/PPTX 文件 | 保存页面、可复用结构、媒体、备注、批注、关系和演示行为的数字包 |

Microsoft 将 PowerPoint 描述为一种可以从空白或模板创建演示、组合文字、图片、艺术元素与视频、添加转场和动画并与他人共享成果的工具。仅从这份能力列表就能看出，演示文稿并不是一叠静态页面。参见 [Microsoft 的 PowerPoint 简介](https://support.microsoft.com/en-us/powerpoint/training/what-is-powerpoint)。

在本文中，**PPT 指 PowerPoint 这一类演示文稿产物及其沟通用途**，不只是旧版 `.ppt` 扩展名。

### 1.1 常见文件类型

| 扩展名 | 用途 |
|---|---|
| `.ppt` | PowerPoint 97–2003 的旧版二进制演示文稿 |
| `.pptx` | 当前无宏 PowerPoint 演示文稿包，也是现代 PowerPoint 主要使用的可编辑格式 |
| `.pptm` | 含宏的 PowerPoint 演示文稿 |
| `.potx` | 可复用的 PowerPoint 模板 |
| `.ppsx` | 打开后直接进入幻灯片放映视图的 PowerPoint 放映文件 |

当前格式及用途见 [Microsoft 文件格式说明](https://support.microsoft.com/en-us/powerpoint/file-formats-that-are-supported-in-powerpoint)。现代 `.pptx` 是 Office Open XML 包，其词汇和打包方式由 [ECMA-376 / ISO/IEC 29500](https://ecma-international.org/publications-and-standards/standards/ecma-376/) 标准化。

## 2. 人们为什么制作演示文稿

人们通常不以“得到一些页面”为最终目的。他们希望改变受众所知道、理解、相信、决定或采取的行动，并且经常需要留下一个组织可以继续使用的成果。

```text
源材料 + 意图
→ 有顺序的论证或解释
→ 共同的视觉体验
→ 受众结果
+ 可编辑、可传播的记录
```

| 沟通任务 | 演示结束后希望发生的变化 |
|---|---|
| 告知 | 受众知道此前不知道的事实 |
| 解释 | 受众理解某种机制、关系或原因 |
| 说服 | 受众接受或认真考虑某个立场 |
| 决策 | 群体从多个选项走向明确选择 |
| 对齐 | 参与者形成共同优先级、语言和下一步 |
| 教学 | 学习者能够理解、记忆或执行某件事 |
| 汇报与问责 | 利益相关者能够评价进展、证据、风险和责任 |
| 动员 | 人们从意识到问题转向采取行动 |
| 留档与交接 | 组织获得可审批、可审计、可复用的正式成果 |

一份 deck 可以同时承担多种任务，但仍然需要一个主要任务。例如，“进展汇报”表面上是在告知，真正目的可能是获得决策、暴露风险或明确责任。

组织沟通研究把提案、会议、报告等反复出现的形式视为组织针对反复出现的场景和目的形成的社会惯例。这个视角对 PPT 很重要：deck 不是中性容器，而是组织沟通实践的一部分。参见 Yates 和 Orlikowski 的 [《组织沟通类型》](https://doi.org/10.5465/amr.1992.4279545)。

## 3. 为什么使用 PowerPoint，而不是其他媒介

当一项任务同时需要下列多种属性时，PowerPoint 具有独特价值：

| 属性 | 价值 |
|---|---|
| 顺序 | 作者可以控制或建议注意力展开顺序 |
| 模块化 | 页面可以重排、隐藏、复制、替换和复用 |
| 多媒体组合 | 文字、图示、数据、图片、音频、视频和运动可以共享同一画布 |
| 演讲支持 | 备注、演讲者视图、计时和导航支持现场传递 |
| 阅读支持 | 阅读视图、批注、备注页、讲义和共享支持异步审阅 |
| 可编辑性 | 产物可以修改、本地化、扩展和改作他用 |
| 组织可移植性 | 同一产物可以穿过会议、评审、审批、归档和复用环节 |

PowerPoint 不应该成为所有信息任务的默认选择：

| 首要需求 | 通常更适合的主要媒介 |
|---|---|
| 很长、线性、可独立阅读的完整论证 | 文档或报告 |
| 动态数据的实时探索 | 电子表格、Notebook 或仪表盘 |
| 开放式群体探索 | 白板或协作画布 |
| 不需要编辑的固定重复播放 | 视频 |
| 交互导航和丰富用户输入 | Web 应用 |
| 有顺序、可视化、可编辑并且需要继续流转的沟通 | PowerPoint |

因此，媒介选择必须由场景决定。“这能不能做成几页 PPT”不如“这个任务是否需要一份有顺序、模块化、可编辑的演示文稿产物”有价值。

## 4. 一份 PPT 有不止一种生命

一份 deck 经常在生命周期的不同阶段承担不同作用：

| 阶段 | Deck 的作用 |
|---|---|
| 活动前 | 思考、综合、评审和排练工具 |
| 活动中 | 协调注意力和节奏的共同视觉界面 |
| 活动后 | 阅读副本、决策记录、讲义、审批材料或复用来源 |
| 没有现场活动 | 异步阅读、录制、旁白或自动播放的演示内容 |

PowerPoint 本身提供不同的编辑、放映、演讲者、阅读、备注、讲义和母版视图。Microsoft 明确建议现场演示时使用带私有备注的演讲者视图，没有演讲者时使用阅读视图查看演示。参见 [为任务选择正确视图](https://support.microsoft.com/en-US/PowerPoint/training/choose-the-right-view-for-the-task-in-powerpoint)。

### 4.1 传递场景会改变设计

| 主要场景 | Deck 应优化的内容 |
|---|---|
| 演讲者主导 | 远距离可读性、节奏、视觉锚点以及对口头表达的补充 |
| 读者主导 | 自解释逻辑、明确上下文、更完整证据和可导航性 |
| 混合场景 | 明确主要模式，并用备注、附录或补充细节服务次要模式 |
| 录制或自动播放 | 旁白、计时、转场、播放可靠性以及不依赖现场讲者 |

不能假设同一页面可以同时最优地服务四种场景。稀疏的演讲者主导页面可能无法独立承担决策记录；密集的董事会材料可能无法在大屏幕上有效传递。必须先明确主要传递场景，再决定信息密度和字体。

Microsoft 对[演讲者备注和讲义](https://support.microsoft.com/en-US/PowerPoint/print-your-handouts-notes-or-slides)、[旁白和计时录制](https://support.microsoft.com/en-us/PowerPoint/training/record-a-slide-show-with-narration-and-slide-timings)以及[共享与共同创作](https://support.microsoft.com/en-us/powerpoint/training/share-your-powerpoint-presentation-with-others)的支持，也说明 PowerPoint 产物横跨现场表达、独立阅读、传播和复用。

## 5. PowerPoint 原生对象模型

`.pptx` 是由相互关联的部件组成的包，不是每页一张位图，也不是一个巨大的单体 XML。Microsoft 的 [PresentationML 结构指南](https://learn.microsoft.com/en-us/office/open-xml/presentation/structure-of-a-presentationml-document)把 Presentation、Theme、Slide Master、Slide Layout 和 Slide 列为核心部件，并允许备注、讲义、批注、媒体、转场和动画等附加部件。

```text
Presentation package
├── Theme
├── Slide Master A
│   ├── Slide Layout A1
│   │   ├── Slide 1
│   │   └── Slide 4
│   └── Slide Layout A2
│       └── Slide 2
├── Slide Master B
│   └── Slide Layout B1
│       └── Slide 7
├── Notes / Handouts / Comments
└── Media / Charts / Tables / Transitions / Animation / Audio
```

| 对象 | 职责 |
|---|---|
| Theme | 可复用的颜色、字体和效果 |
| Slide Master | 一组 Layout 共同继承的格式和对象 |
| Slide Layout | 某类页面的默认外观、位置和占位符 |
| Placeholder | Layout 上预先格式化、带类型的内容容器 |
| Slide | 绑定某个 Layout 的一次具体内容实例 |
| Notes 与 Handouts | 可见页面之外的演讲者和受众通道 |
| Package relationships | 连接页面、Layout、Master、媒体、备注、批注及其他部件 |
| Presentation behavior | 顺序、转场、动画、旁白和计时 |

Microsoft 把 Slide Layout 定义为承载格式、位置和占位符的结构，并明确显示其位于 Slide Master 之下。参见 [什么是 Slide Layout](https://support.microsoft.com/en-US/PowerPoint/what-is-a-slide-layout)。

这个层级非常重要，因为视觉上正确的截图仍然可能是一份结构很差的 PowerPoint。真正可用的 deck 还需要合理的对象边界、继承关系、可编辑性、包关系和演示行为。

## 6. 什么是 PowerPoint 模板

Microsoft 区分 Theme 和 Template：Theme 提供协调的颜色、字体和效果；Template 则在 Theme 之上增加面向特定用途的内容和结构。参见 [Theme 与 Template 的区别](https://support.microsoft.com/en-us/powerpoint/understand-the-difference-between-powerpoint-templates-and-themes)。

由此可以得到一个实用原则：

> 模板之所以存在，是因为某种演示身份、结构或沟通场景预计会重复出现。

在实际使用中，一个 PowerPoint 模板可以组合多个可复用层次：

| 层次 | 提供的内容 |
|---|---|
| Theme | 协调一致的颜色、字体和效果 |
| Slide Master | 一组 Layout 共同使用的格式和固定对象 |
| Slide Layout | 某类重复页面的占位符结构和默认构图 |
| 预置内容与示例 | 面向特定用途的起始页面、固定措辞、示例和使用指引 |

模板可以减少重复设置、保持视觉与结构一致、编码反复出现的演示模式，并让后续编辑和评审更加可预期。一份 deck 可以被复制或作为参考，但视觉相似本身并不会自动形成结构良好的模板。有效模板需要把稳定、可复用的决策与只属于某一次演示的内容分开。

## 7. 什么是好的 PPT

好的 deck 不能只用视觉精美来定义。质量至少包含以下层次：

| 层次 | 核心问题 |
|---|---|
| 事实 | 论断、数值、来源和概念区分是否正确？ |
| 沟通 | 受众、预期结果、核心信息和行动要求是否明确？ |
| 叙事 | 页面顺序是否形成预期的理解、判断或决策？ |
| 认知 | 人们是否能在不过载的情况下感知、处理并连接信息？ |
| 视觉 | 层级、字体、留白、图像和数据图形是否服务于含义？ |
| 原生与运行 | Deck 是否能够可靠地演示、编辑、评审、复用和交付？ |

研究并不支持“图片越多越好”这样的普遍规则。Mayer 对多媒体学习研究的综合指出，设计良好的文字与图形组合可以改善理解，同时人的信息处理容量有限。参见 [《多媒体学习》](https://www.cambridge.org/highereducation/books/multimedia-learning/FB7E79A165D24D47CEACEB4D2C426ECD)。

Bartsch 和 Cobern 发现，即使 PowerPoint 本身可能有帮助，不相关的图片和声音仍会损害记忆与学习。参见 [《课堂中 PowerPoint 演示的有效性》](https://doi.org/10.1016/S0360-1315(03)00027-7)。Kosslyn 等人发现，真实演示中经常违反可辨别性、有限容量和信息变化原则，而且未经训练的观察者常常无法识别这些设计问题。参见 [《PowerPoint 演示的缺陷与失败：心理学分析》](https://doi.org/10.3389/fpsyg.2012.00230)。

结论不是建立一个万能页面公式，而是一条由场景决定的质量原则：

> 每个元素都应该帮助目标受众，在目标传递场景中完成目标认知或组织任务。

## 8. PPT 不是什么

PPT 并不天然等于：

- 把一份文档分页后加上装饰；
- 一组漂亮但互不相关的画布；
- 演讲者将要说的话的逐字稿；
- 视觉效果展示；
- 没有后续生命的一次性放映；
- 只是带有 `.pptx` 扩展名的扁平图片；
- 因为页面看起来一致就自动成为模板；
- 所有沟通任务的正确媒介。

如果用户从软件默认提供的页面形式出发，而不是从受众真正要完成的任务出发，PowerPoint 的惯例也可能反过来扭曲沟通。工具应该服务沟通模型，而不是根据工具按钮倒推沟通模型。

## 9. 每个 PPT 请求都应该回答的问题

| 问题 | 为什么重要 |
|---|---|
| 受众是谁？ | 决定语言、默认知识、证据和语气 |
| 看完后必须发生什么变化？ | 定义沟通任务和成功条件 |
| 主要是现场讲、独立读、录制还是混合？ | 决定密度、字体、备注和演示行为 |
| 核心信息是什么？ | 防止在没有论证时直接生产页面 |
| 哪些证据必须可见或可追溯？ | 保护事实完整性和决策价值 |
| 什么可以改变，什么必须保留？ | 明确编辑边界和评审预期 |
| 哪些内容必须可编辑或保持原生？ | 定义对象模型和交付要求 |
| 什么会反复出现？ | 决定是否需要 Theme、Master、Layout、预置内容，或根本不需要模板 |

这些问题应该先于对风格、版式、模板、动画或制作方式的详细决定。
