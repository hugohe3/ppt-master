# 生成决策逻辑（像 PPT 专家一样思考）

> 这份文档把分散在 `SKILL.md` / `strategist.md` / `design_spec_reference.md` / `spec_lock` 各处的决策，串成**一条显式的"专家决策流"**：输入是什么、什么时候用配图、什么时候手画 SVG、什么时候放单页素材、什么时候排版/确认。它是导航，不替代各文件的权威细节。

---

## 0. 一句话心智模型

> **先把整本规划清楚 → 一次确认 → 再逐页生产。**

和真实 PPT 专家一样：拿到材料先理结构、定每页怎么呈现、定全本风格，方向对齐后才动手做。**Strategist 做完所有决策并锁进 `spec_lock.md`，Executor 只是按锁去执行。**

---

## 1. 输入是什么

| 输入 | 必需？ | 说明 |
|---|---|---|
| **源材料** | 必需 | PDF / DOCX / URL / Markdown / 一段文字 → 先转成 Markdown，作为"真相源"。**这是真正的输入。** |
| 风格提示 | 可选 | "麦肯锡风格 / 极简 / 科技蓝"等 → 进八确认 d–g 当风格简报 |
| 模板路径 | 可选 | 显式给 brand/layout/deck 目录路径才触发模板 |

> **大纲、章节结构不是输入——是 Strategist 读源材料后推导出来的产物**（design_spec §IX）。只给主题没有材料时，先跑 `topic-research` 抓料，再进流程。

---

## 2. 关键澄清：SVG 是底板，不是"选项之一"

**每一页本身永远是一张手写的 SVG 画布。** "配图 / 图表 / native diagram"都是**填进这张 SVG 的内容载体**，不是和 SVG 二选一：

```
一页 = SVG 画布
        ├─ 手画矢量（标题/正文/卡片/线条…）          ← 永远有
        ├─ [可选] 配图区域  <image>                   ← 氛围/场景/封面
        ├─ [可选] 图表       继承 charts/ 模板重画
        └─ [可选] 原生图示   <rect data-native-diagram> ← 精致立体结构图
```

所以正确的问法不是"用图还是用 SVG"，而是：**这页的 SVG 里，该填哪些载体。**

## 3. 专家决策流（阶段顺序）

```
源材料
  │
  ▼  ┌─────────────────────── Strategist：规划 + 锁定 ───────────────────────┐
  │  │ 1. 读材料 → 推导结构：分几章、多少页、每页的"核心信息"(§IX Core message)│
  │  │ 2. 逐页"载体决策"（见 §4）→ 填 §VII 可视化 / §VIII 配图清单            │
  │  │ 3. 全本锁定：配色(§III) 排版(§IV) 图标(§VI) 配图风格锁(§III AI 策略)    │
  │  │ 4. 写 design_spec.md + spec_lock.md（颜色/字体/页面节奏/page_charts/   │
  │  │    page_diagrams/图片清单 都落进机器契约）                             │
  │  └──────────────────────────────────────────────────────────────────────┘
  │
  ▼  ⛔ 八确认（唯一阻塞门）：画布/页数/受众/风格/配色/图标/排版/配图 一次性确认
  │
  ▼  [配图阶段] §VIII 里有 ai/web 行才触发：生成/搜图
  │
  ▼  ┌─────────────────────── Executor：逐页生产 ───────────────────────┐
  │  │ 每页：重读 spec_lock → 取该页 节奏/模板/图表/原生图示/图片           │
  │  │ → 手写这页 SVG（按锁定的色/字/图标），把载体填进去/写占位符         │
  │  └──────────────────────────────────────────────────────────────────┘
  │
  ▼  质检 → 后处理 → svg_to_pptx：逐元素转原生 DrawingML，解析所有占位符
  │
  ▼  可编辑 PPTX
```

**注意决策都在确认门之前完成**——这正是"先规划后生产"。

## 4. 每页的"载体决策"（核心，专家逐页判断）

对每一页，按它的**核心信息形态**决定 SVG 里填什么。问四个问题：

### Q1. 这页的信息是"结构关系"吗？（流程/层级/对比/构成/循环/汇聚…）
→ 是：需要一个**结构图载体**。再按 **内容 × 全本风格** 三选一：

| 选 | 条件 |
|---|---|
| **原生图示** `native diagram` | 关系能对上某个精品图示，且**全本风格接受立体感**（八确认 d）。少而精、点睛用。按 scenario→type→slots→footprint→density→motif 选，缩略图终选 → 记入 `page_diagrams` |
| **`charts/` SVG 模板** | 扁平/极简全本，或要数据图，或要改结构（增删项）。继承模板重画 → 记入 `page_charts` |
| **手画 SVG** | 没有合适模板，要完全定制版式 |

### Q2. 这页需要"氛围/场景/人物/封面"吗？（矢量画不出的视觉）
→ 是：加一条**配图**（§VIII）。再分：
- `page_role: hero_page` —— 图就是这页主体（封面/章节页/情绪转场/大数字/金句）。
- `page_role: local` —— 图只占一块区域，SVG 在旁边/上面承载信息。
- `Acquire Via`：`ai`（生成）/ `web`（搜图）/ `user`（用户给）/ `formula`（公式渲染）。

### Q3. 这页是"概念/文字/要点"？
→ 直接**手画 SVG 版式**（§V 版式库 + 该页 `page_rhythm`：anchor/dense/breathing）。这是默认主体。

### Q4. 载体会叠加吗？（专家常用）
→ 会。典型："配图铺底 + 原生图示/图表叠在上面 + SVG 文字"——image-as-canvas 家族。规划时就把这页的**载体栈**想清楚（§VIII 的 Layout pattern 列 + §VII 选择共同表达）。

> **图标**贯穿所有载体，按需 `ls templates/icons/<库> | grep` 取用，全本一个风格库。

## 5. "什么时候做什么"对照表

| 决策 | 什么时候 | 在哪记录 |
|---|---|---|
| 用**生成配图** | 规划期，页面要氛围/场景/封面/人物 | §VIII `Acquire Via: ai/web` |
| 用**图表模板** | 规划期，页面是数据/结构关系且扁平风 | §VII → spec_lock `page_charts` |
| 用**原生图示** | 规划期，关系适配精品图且接受立体风 | §VII → spec_lock `page_diagrams` |
| 用**手画 SVG** | 永远（页面底板）；纯概念/文字页全靠它 | §IX Layout + §V |
| **排版**（版式） | 全本规划期定原则(§V)，逐页定节奏(`page_rhythm`)/模板(`page_layouts`)；生产期落到每页 | §V / spec_lock |
| **排版**（字体字号） | 规划期全本锁定 | §IV → spec_lock `typography` |
| **确认** | 规划全部完成后，**唯一一次**八确认 | 八确认门 |
| **生产** | 确认后，逐页手写 SVG | Executor |

## 6. 一致性怎么保证（四类载体共存不打架）

全部载体最终都**归一到 deck 的同一套调色板与字体**，各用各的机制：

| 载体 | 归一机制 |
|---|---|
| 手画 SVG / 图表模板 | Executor 每页重读 `spec_lock` 取色/字（抗长稿漂移） |
| 配图 | 三维锁（rendering × palette × type）整本锁定 + prompt 注入 HEX |
| 原生图示 | 占位符 `data-recolor` 把基色映射到 `spec_lock` 主/辅色 |
| 图标 | 全本一个风格库 + `fill` 上色 |

## 7. 已知的待改进（诚实记录）

- **载体决策目前是碎片化的**：配图(§VIII) 与 结构图(§VII) 在不同小节分别填，没有一个统一的"这页最佳载体"决策节点。本文档是把它**显式串起来的视图**；若要在提示词层也归一，可在 `strategist.md` 增设一步"逐页载体决策"，把 §VII/§VIII 收敛成一次整体判断。
- 原生图示选材最终仍需**看缩略图终选**（标签缩小到候选，眼睛定夺）——这是视觉素材库的固有特性，非缺陷。
