# 项目定位与能力边界

[English](../project-positioning.md) | [中文](./project-positioning.md)

---

本文定义 PPT Master 的长期产品定位，以及新增、保留或削减能力时使用的判断框架。它不是功能清单，也不是执行手册。

本文是[英文规范源](../project-positioning.md)的同步中文译本，不形成第二套
产品政策；定位变化必须在同一次修改中同步两种语言，如有歧义以英文规范
源为准。

当前具体如何选择路线，仍以 [`workflows/routing.md`](../../skills/ppt-master/workflows/routing.md) 为准。本文回答的是另一个问题：一个方向究竟是否应该属于 PPT Master。

## 一、项目定位

> **PPT Master 是一套面向 AI Agent、以对话驱动的开源演示文稿创作框架。它通过明确的创作与保留契约，把主题、源材料、设计参考或已有 PowerPoint，转化为高质量、原生可编辑且由用户本地持有的 PowerPoint 产物。**

PPT Master 不是模型，不是演示 SaaS，也不是 PowerPoint 的替代品。它是通用 AI Agent 与演示文稿领域之间的 harness：工作流提供专业判断和执行纪律，确定性工具负责转换、校验、打包以及可重复的文件操作。

最主要的交付物仍然是用户能够继续编辑的 PowerPoint 文件。可复用模板工作区、项目源材料、设计规范、预览和校验产物同样是一等支撑产物，因为它们让最终 deck 可控制、可重建、可复用。

## 二、PPT Master 解决的问题

制作一份真正有用的演示文稿，不是一次文件格式转换。它包含多种工作，而通用 AI 和现有演示工具通常无法同时把这些工作做好。

| 问题 | PPT Master 的处理方式 |
|---|---|
| 通用 AI 能理解材料，却无法稳定构造复杂 PowerPoint 文件 | 为 Agent 提供演示文稿专用的策划、设计、创作和校验契约 |
| 视觉效果较好的 AI 产物经常是整页图片或网页文档 | 使用受约束的视觉创作格式编译真实 PowerPoint 对象，不把整页压平 |
| 直接生成 OOXML 过于冗长、脆弱，不适合作为 AI 的视觉创作语言 | 让 Agent 使用模型友好的视觉语言创作，再由确定性编译器和 package 校验器完成交付 |
| 源事实、叙事结构、视觉设计、素材和最终交付通常彼此割裂 | 用统一项目生命周期和明确的产物所有权连接各阶段 |
| 已有 deck 可能需要重新设计、替换内容、复用设计或只追加演示行为 | 根据用户意图选择具有相应保留契约的修改模型 |
| 托管式演示工具带来成本、隐私和平台锁定 | 保持开源、本地优先、模型无关和 Agent 无关 |

因此，项目的工作不只是“写出一个 `.pptx`”，而是让通用 AI Agent 具备可靠完成演示文稿工作的能力，同时保留用户检查、编辑和拥有结果的权利。

## 三、目标用户与使用方式

PPT Master 主要服务于以下用户：

- 手上有主题、文档、数据、视觉参考、品牌资产或已有 deck，需要把它们转化为演示文稿。
- 相比几秒钟出片，更重视设计质量和原生可编辑性。
- 需要本地持有项目、透明控制成本，并保留选择 AI Agent 和模型的自由。
- 愿意确认关键方向，并在必要时继续用 PowerPoint 精修生成结果。
- 能够使用对话式 AI IDE 和本地 Python 环境，即使本人并不编写代码。

PPT Master 不以零配置浏览器出片、实时团队共编，或完全不需要人类判断和修改的一次性完稿为主要目标。

## 四、产品承诺

| 承诺 | 含义 | 边界 |
|---|---|---|
| 原生可编辑 | 在所选路线支持的范围内，将文字、形状、图片、图表、表格和演示行为保留为真实 PowerPoint 构造 | 不把整页截图作为规范 PPTX 输出；无法表达的语义必须显式说明 |
| 高质量起点 | 消除从原材料到结构清晰、经过设计的 deck 的大部分工作 | 不承诺一次生成完美终稿；模型能力和用户判断仍决定上限 |
| 事实与意图保真 | 区分来源事实、用户决策、设计建议和派生产物 | 不编造依据，也不把保留型请求静默改成重新设计 |
| 用户所有权 | 产物保持本地、可迁移，不依附单一厂商产品 | 用户选择的模型、搜索、生图或语音服务仍可能发生外部调用 |
| 工程可靠性 | 使用明确路线、执行契约、质量门、回读校验和可恢复项目状态 | 不用静默降级掩盖失败，也不发布未通过必需 gate 的产物 |
| 质量优先 | 优先保证整份 deck 的一致性、可编辑性和交付可靠性 | 可以在不损害质量时提效，但不默认采用低质量并行生成 |

## 五、产品能力模型

PPT Master 的能力范围应当比单一生成管线更宽，但比通用办公 Agent 更窄。

| 能力领域 | 项目责任 |
|---|---|
| 演示理解与策划 | 把主题或材料包转化为面向受众的核心信息、叙事结构、页面规划和明确设计方向 |
| 新演示文稿创作 | 创作新的页面视觉，并编译为原生可编辑 PowerPoint |
| 可复用演示设计系统 | 创建、组合、校验并应用 Brand、Layout 和 Deck 工作区 |
| 已有 deck 改编 | 在不同保留契约下重做已有 deck、向原生页面壳填入新内容，或追加原生演示行为 |
| 演示表达 | 在沟通目标需要时使用图片、图示、图表、表格、公式、讲稿、旁白、转场和动画 |
| 审阅与交付 | 预览、检查、校验、修复、导出、回读，并保留足够的项目状态供后续精修或重新导出 |

这些是产品责任，不代表每项责任都必须暴露成顶层路线或单独的 workflow 文件。只有输入、修改规则、不变量或产物生命周期确实不同，才需要独立路线。

## 六、项目拥有的责任与接入的能力

PPT Master 可以依赖通用能力，但不因此成为这些通用能力的平台。

| 领域 | PPT Master 负责 | PPT Master 不负责 |
|---|---|---|
| 研究 | 判断 deck 需要什么证据、保存来源，并把研究结果转化为演示内容 | 与演示任务无关的通用网络研究引擎 |
| 图片 | 判断是否需要图片，以及图片的角色、风格、来源、位置、出处和就绪状态 | 通用图片生成或图片管理平台 |
| 音频 | 演示语境中的讲稿、音色选择、逐页旁白、计时和 PowerPoint 嵌入 | 通用音频工作站、播客平台或语音供应商市场 |
| 数据可视化 | 选择表达形式、保留数据、校验几何，并决定可编辑性与保真度之间的取舍 | 通用 BI 或电子表格产品 |
| 模板与品牌 | 明确且可复用的演示身份、版式、Master/Layout、槽位、素材与组合契约 | 恢复来源文件中已经不存在的历史设计意图 |
| PowerPoint 编辑 | 演示专用的生成、填充、重做和有边界的原生增强 | 替代 PowerPoint 的完整编辑界面，或支持任意 OOXML 修改 |

仓库可以内置多个 provider，以保持开放性和实际可用性，但供应商特异行为应当位于稳定的接入边界之后。演示工作流负责选择逻辑和输出语义，任何单一 provider 都不应重新定义产品边界。

## 七、稳定的技术策略

技术架构服务于定位，但技术架构本身不是项目定位。

- **受约束的 SVG → DrawingML** 是新设计页面的主要创作和编译路线。
- **直接 OOXML 操作**用于必须保留已有 PowerPoint package 的场景。
- **模板工作区**在新页面创作前声明可复用设计结构，而不是在成品形成后反向推断。
- **项目产物和质量门**让过程可以检查、续跑和验证。

具体实现可以演进，但以下五条不变量应保持稳定：

1. 不把规范 deck 扁平化成每页一张图片。
2. 明确说明可编辑性、视觉保真和保留程度之间的取舍。
3. 修改产物前先确定创作或修改契约。
4. 区分来源、作者状态、派生状态和交付状态。
5. 必需语义无法安全表达时应当停止，不声称并不存在的保真度。

## 八、明确不做

PPT Master 不以成为以下产品为目标：

- 完全替代人类演示判断的全自动系统。
- 通用办公助手、通用研究平台、通用图片平台或通用音频平台。
- 完整的 PowerPoint 克隆、完整浏览器自由画布编辑器、实时协作服务或演示 SaaS。
- 任意 SVG 到 PPTX 或任意 OOXML 的转换服务。
- 从完成态 PPTX 或 SVG 中恢复缺失的历史 Master/Layout 意图。
- 把推断出的模板结构原地嫁接到已有文件上的升级器。
- 以牺牲 deck 一致性或交付可靠性为代价的速度优先生成器。

这些非目标并不排斥在演示任务中使用研究、图片、音频、原生对象或已有 deck。它们防止支撑能力脱离演示语境，发展成承诺完全不同的独立产品。

## 九、能力准入与削减判据

评估任何新增能力时，按以下顺序回答：

1. **用户任务**：它完成了哪一个真实的演示文稿任务？
2. **责任结果**：它创建或保护了什么演示产物、决策或质量属性？
3. **不变量**：什么必须保持不变，什么允许改变？
4. **产品层级**：它属于核心能力、演示专用扩展、可替换 provider adapter，还是仓库维护？
5. **可验证性**：能否明确检查成功与失败，而不是依赖模糊承诺？
6. **真实证据**：是否有真实用户需求、重复工作流或已经发生的失败，足以覆盖维护成本？

| 决策 | 适用条件 |
|---|---|
| 作为核心能力新增或保留 | 直接完成演示任务，并需要演示专用契约或校验 |
| 作为集成扩展保留 | 能力本身可选，但它的规划和输出语义是演示专用的 |
| 放到稳定接入边界之后 | 底层服务具有通用性或供应商特异性，PPT Master 负责演示专用的选择逻辑和输出契约 |
| 移到仓库工具层 | 服务于仓库、示例、安装或贡献者流程，而不是生产演示文稿 |
| 退役 | 与另一权威重复、没有有效使用者、承诺无法验证，或维护成本高于演示价值 |

文件或 workflow 的数量本身，不是新增或删除能力的理由。真正的判断标准是责任是否清晰，以及它是否创造明确的产品价值。

## 十、北极星结果

一次成功的 PPT Master 使用过程应当是：

> 用户把主题、源材料、设计参考、可复用模板或已有 PowerPoint 交给 AI Agent，确认重要的内容和设计选择，最终获得一份结构连贯、原生可编辑、经过校验的 PowerPoint 产物，以及足够的本地项目状态，用于审阅、精修、复用或重新导出。

未来工作应按以下顺序改善这一结果：

1. 输出正确性、原生可编辑性和交付可靠性。
2. 内容推理、叙事质量和视觉一致性。
3. 品牌、版式、Deck 和已有 PowerPoint 资产的复用。
4. 人工审阅、修正和可控迭代。
5. 能够强化前四项的额外格式、provider 和便利能力。
