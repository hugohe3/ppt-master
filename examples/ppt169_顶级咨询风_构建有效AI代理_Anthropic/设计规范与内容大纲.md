## 设计风格：顶级咨询

### 核心设计原则

1. **结论先行 (Pyramid Principle)**: 每页标题即核心观点
2. **数据情境化**: 对比、趋势、基准，不孤立呈现数据
3. **SCQA 框架**: 情境→复杂性→问题→答案
4. **MECE 原则**: 相互独立、完全穷尽
5. **专业留白**: 内容占比 < 65%，让信息"呼吸"

### P04: What are Agents - 定义与分类

**标题**: "Agentic Systems = Workflows + Agents"

| Workflows                                  | Agents                                  |
| ------------------------------------------ | --------------------------------------- |

### P06: Building Block - 增强型 LLM

**布局**: 中央图示 + 周边说明

**标题**: "基础构建模块：增强型 LLM (Augmented LLM)"

**内容**:

- 中央: LLM 核心
- 三个增强模块:
  1. **Retrieval (检索)**: 生成搜索查询，获取上下文
  2. **Tools (工具)**: 选择并调用适当工具

**关键建议**:

### P07: Workflow 1 - Prompt Chaining

**布局**: 水平流程图 + 说明

**标题**: "Workflow 模式 1: Prompt Chaining (提示链)"

**内容**:

```
[LLM 1] → Gate ✓ → [LLM 2] → Gate ✓ → [LLM 3] → Output
```

**何时使用**:

- 任务可清晰分解为固定子任务
- 用延迟换取更高准确性

**示例**:

- 生成营销文案 → 翻译成其他语言
- 写大纲 → 检查标准 → 基于大纲写文档

**设计要点**:

- 水平箭头流程图
- Gate 用菱形表示
- 配图标: `arrow-right`, `check-circle`

### P08: Workflow 2 - Routing

**布局**: 分支流程图

**标题**: "Workflow 模式 2: Routing (路由分发)"

**内容**:

```
        Input
          ↓
      [分类器]
     ↙   ↓   ↘
 [专业A] [专业B] [专业C]
```

**何时使用**:

- 复杂任务有明确类别，需分别处理
- 分类可被准确执行

**示例**:

- 客服查询分流: 常规问题 / 退款请求 / 技术支持
- 模型分流: 简单问题 → Haiku, 复杂问题 → Sonnet

**设计要点**:

- 树形分支图，从上到下
- 不同分支用不同颜色

### P09: Workflow 3 - Parallelization

**布局**: 并行流程 + 两种变体

**标题**: "Workflow 模式 3: Parallelization (并行化)"

**内容**:

**两种变体**:

| Sectioning (分区)             | Voting (投票)                |
| ----------------------------- | ---------------------------- |
| 任务拆分为独立子任务并行执行  | 同一任务多次执行获取多样输出 |
| 示例: 护栏检测 + 核心响应分离 | 示例: 多角度代码漏洞审查     |

**何时使用**:

- 子任务可并行以加速
- 需要多视角或多次尝试以提高置信度

**设计要点**:

- 并行箭头图示
- 左右分栏展示两种变体

### P10: Workflow 4 & 5 - Orchestrator & Evaluator

**布局**: 上下分区，两种模式

**标题**: "Workflow 模式 4 & 5: 动态编排与迭代优化"

**内容**:

**Orchestrator-Workers (编排者-工作者)**:

```
[Orchestrator LLM] 动态分解任务
      ↓ ↓ ↓
[Worker 1] [Worker 2] [Worker N]
      ↓ ↓ ↓
   [综合结果]
```

- 适用: 无法预测子任务的复杂任务（如多文件代码修改）

**Evaluator-Optimizer (评估者-优化器)**:

```
[Generator LLM] ⟷ [Evaluator LLM]
     生成响应     提供反馈
       ↻ 迭代循环
```

- 适用: 有明确评估标准，迭代改进有价值（如文学翻译）

**设计要点**:

- 上下分区，每区一个模式
- 使用循环箭头表示迭代

### P11: Autonomous Agents - 自主代理

**布局**: 循环流程 + 关键特征

**标题**: "Agents: LLM 在循环中基于环境反馈使用工具"

**内容**:

```
Human Input
     ↓
   [LLM]
   ↙   ↘
[Tool Use] → [Environment]
   ↖   ↗
  Feedback
     ↓
Human Checkpoint / Completion
```

**关键特征**:

1. **任务来自人类**: 命令或交互讨论
2. **自主规划与执行**: 独立操作，必要时返回人类
3. **环境反馈**: 工具调用结果、代码执行结果
4. **检查点**: 人类反馈或阻塞时暂停
5. **终止条件**: 完成或达到最大迭代次数

**设计要点**:

- 使用 `agent_loop.png` 或手绘循环图
- 强调人类监督节点

### P12: When to Use Agents - Agent 适用场景

**布局**: 适用条件 + 风险提示

**标题**: "Agent 适用于开放式问题，但需充分测试与防护"

**内容**:

**✅ 适用条件**:

- 无法预测所需步骤数
- 无法硬编码固定路径
- 需信任 LLM 决策能力
- 在受信环境中扩展任务

**⚠️ 风险提示**:

- 更高成本
- 错误可能复合累积
- 建议: 沙盒环境广泛测试 + 适当防护措施

**Anthropic 实践案例**:

- SWE-bench 编码代理
- Computer Use 参考实现

**设计要点**:

- 左右分栏: 适用 vs 风险
- 使用 `check-circle` 和 `warning` 图标

### P13: Case Studies - 实践案例

**布局**: 双列案例卡片

**标题**: "Agent 实践领域: 客户支持 & 编程代理"

**内容**:

**A. 客户支持 (Customer Support)**

- 自然对话流 + 工具集成
- 拉取客户数据、订单历史、知识库
- 执行退款、更新工单

**B. 编程代理 (Coding Agents)**

- 代码解决方案可通过自动化测试验证
- 使用测试结果作为反馈迭代
- 问题空间明确且结构化
- 输出质量可客观衡量

**设计要点**:

- 两个等宽卡片
- 配图标: `users`, `code`

### P14: Tool Engineering - 工具设计

**布局**: 原则列表 + 示例

**标题**: "Prompt Engineer Your Tools: 工具设计与 HCI 同等重要"

**内容**:

**核心原则 (ACI = Agent-Computer Interface)**:

1. **给 LLM 足够的 token "思考"**

   - 避免在写输出前陷入死角
2. **格式贴近自然文本**

   - Markdown > JSON（减少转义开销）
   - Diff 比重写更难（需预知行数）
3. **无格式开销**

   - 不要求精确计数千行代码
   - 不要求手动字符串转义
4. **站在模型角度思考**

   - 工具定义是否够清晰？
   - 包含示例用法、边界情况、输入格式
5. **Poka-yoke (防错设计)**

   - 示例: 使用绝对路径替代相对路径

**设计要点**:

- 编号列表，每条配说明
- 突出 "我们在工具优化上花的时间比整体 prompt 更多"

### P15: Summary - 核心原则

**布局**: 三大原则 + 结语

**标题**: "构建正确的系统，而非最复杂的系统"

**内容**:

**三大核心原则**:

| # | 原则                   | 说明                             |
| - | ---------------------- | -------------------------------- |
| 1 | **保持简单**     | Agent 设计应简洁，避免过度工程   |
| 2 | **透明可见**     | 显式展示 Agent 的规划步骤        |
| 3 | **精心设计 ACI** | 通过详尽的工具文档和测试打造接口 |

**行动建议**:

- 从简单 prompt 开始
- 用全面评估优化
- 仅在简单方案不足时添加多步 Agent

**结语**:

> "框架可帮助快速起步，但走向生产时不要犹豫减少抽象层，用基础组件构建。"

**设计要点**:

- 三原则用卡片突出
- 结语用引用框样式
- 底部可加 Anthropic logo 和来源链接
