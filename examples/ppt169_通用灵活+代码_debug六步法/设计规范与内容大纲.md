# 设计规范与内容大纲

## 项目信息

- **项目名称**: Debug 六步法 - 高手都做对哪些事
- **画布格式**: PPT 16:9 (1280×720)
- **设计风格**: 通用灵活风格 + 代码/科技感
- **页数**: 10 页
- **目标受众**: 程序员、开发者、技术团队成员
- **使用场景**: 技术团队内部培训/分享会
- **创建日期**: 2025-11-28

## 设计规范

### 1. 画布设置

- **尺寸**: 1280×720
- **viewBox**: `0 0 1280 720`
- **背景**: 使用 `<rect>` 元素

### 2. 配色方案（深色科技风）

| 用途        | 色值      | 说明                |
| ----------- | --------- | ------------------- |
| 背景色      | `#0d1117` | GitHub 深色主题背景 |
| 卡片背景    | `#161b22` | 次级容器背景        |
| 边框/分割线 | `#30363d` | 微妙边界            |
| 主强调色    | `#58a6ff` | 链接蓝/高亮         |
| 次强调色    | `#7ee787` | 成功绿/代码绿       |
| 警告色      | `#f0883e` | 橙色提示            |
| 错误色      | `#f85149` | 红色/Bug 标记       |
| 主文字      | `#e6edf3` | 高对比白            |
| 次文字      | `#8b949e` | 灰色说明            |
| 代码背景    | `#1f2428` | 代码块底色          |

### 3. 字体规范

| 用途      | 字体                          | 字号 | 字重     |
| --------- | ----------------------------- | ---- | -------- |
| 主标题    | SF Pro Display, -apple-system | 48px | Bold     |
| 副标题    | SF Pro Display, -apple-system | 32px | Semibold |
| 步骤编号  | SF Mono, monospace            | 72px | Bold     |
| 正文      | SF Pro Text, -apple-system    | 20px | Regular  |
| 代码/标签 | SF Mono, Menlo, monospace     | 18px | Regular  |
| 小注释    | SF Pro Text                   | 16px | Regular  |

### 4. 布局规范

- **安全边距**: 60px（四周）
- **卡片圆角**: 12px
- **卡片内边距**: 24px
- **元素间距**: 24px（同级）、40px（分组）
- **阴影**: `drop-shadow(0 4px 12px rgba(0,0,0,0.4))`

### 5. 视觉元素

- **步骤编号**: 大号等宽字体 + 渐变色块背景
- **图标风格**: 线性图标，2px 描边，圆角端点
- **代码块**: 深色背景 + 左侧彩色边线
- **流程箭头**: 虚线 + 三角箭头

### 6. 技术约束

- ✅ 使用 `<tspan>` 进行手动换行
- ❌ 禁止使用 `<foreignObject>`
- ✅ 背景使用 `<rect>` 元素
- ✅ 遵循 CRAP 设计原则（对齐、对比、重复、亲密性）

## 内容大纲

### Slide 01 - 封面

- **主标题**: Debug 六步法
- **副标题**: 高手都做对哪些事？
- **装饰**: 代码符号背景 `{ } < > / * `
- **底部**: 技术分享 · 2025

### Slide 02 - 核心理念

- **标题**: 🤖 系统化 Debug 流程
- **核心观点**: "修 Bug 的关键不在于动手快，而在于理解问题准"
- **对比卡片**:
  - ❌ 瞎猜乱试 → 看到错误就慌张
  - ✅ 有条理 → 可控地解决问题
- **视觉**: 左右对比布局

### Slide 03 - 步骤 1

- **步骤编号**: 01
- **标题**: ⏸️ 厘清流程的合理性
- **核心动作**: 暂停 → 用白话描述整个流程
- **案例框**: 购物车结账失败案例
- **灵魂三问卡片**:
  1. 流程本该如此运作吗？
  2. 哪些符合预期？
  3. 是流程矛盾还是代码写错？

### Slide 04 - 步骤 2

- **步骤编号**: 02
- **标题**: 📍 厘清 Bug 在流程中的位置
- **核心观点**: 不要直接查报错信息，要定位转折点
- **流程图示**: 用户触发 → 系统呼叫 → 哪一步没接上？
- **目标框**: 明确指出 "A 与 B 之间" 或 "B 处理 C 时"

### Slide 05 - 步骤 3

- **步骤编号**: 03
- **标题**: 🔍 排除法缩小范围
- **核心观点**: 不要瞎猜，系统化二分法排除
- **三列检查卡片**:
  1. 确认通讯 - Postman 测 API
  2. 检查转换 - 数据在哪里"变质"
  3. 检查输出 - 逻辑错 vs 字段名拼错

### Slide 06 - 步骤 4

- **步骤编号**: 04
- **标题**: 🧪 验证假设
- **核心观点**: 像做实验一样，控制变量
- **方法**: 每次测试只改一个条件
- **案例框**: 画面没更新 → 数据变了但画面没动 → 问题在渲染层

### Slide 07 - 步骤 5

- **步骤编号**: 05
- **标题**: ✅ 确认修正结果与副作用
- **警示**: 修好 A 坏了 B 是最可怕的
- **检查清单**:
  - 对照测试：记录修正前后差异
  - 连带影响：函数被其他模块共用了吗？
  - 可维护性：暴力解法务必写注释

### Slide 08 - 步骤 6

- **步骤编号**: 06
- **标题**: 🔄 全面测试与记录
- **核心动作**: 确保系统回归稳定
- **三列卡片**:
  1. 重现与回归 - 原始问题消失
  2. 边界测试 - 空值、极大值、断网
  3. 文档化 - 记录原因和解法

### Slide 09 - 总结

- **标题**: ✨ Debug 的本质
- **三步流程图**: 先理解 → 再定位 → 后验证
- **金句**: 从"修补匠"进阶为能掌控系统的开发者
- **适用场景**: 手写代码 & AI 辅助开发

### Slide 10 - 封尾

- **主文字**: Thank You
- **副文字**: Q & A
- **装饰**: 与封面呼应的代码符号
