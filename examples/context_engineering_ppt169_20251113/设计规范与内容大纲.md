#### Slide 05: 形式化定义

**标题**：Context Engineering 的形式化定义

**内容结构**：

**核心定义**：

```
CE: (C, T) → f_context

其中：
- C: 原始上下文信息（实体的情境特征化）
- T: 目标任务或应用领域
- f_context: 上下文处理函数
```

**关键组件**：

1. **Entity (实体)**：用户、应用、对象、环境
2. **Characterization (特征化)**：描述实体的信息集合
3. **Context (上下文)**：所有相关实体的特征化并集
4. **Context Engineering**：设计并优化上下文收集、存储、管理和使用的系统过程

**强调**：

> "Context = 任何可用于描述实体情境的信息" — Dey, 2001

**设计要点**：

- 公式居中展示
- 关键术语加粗

**Era 4.0 (Speculative)**: 超人类智能

- 机器智能：超越人类理解
- 上下文形式：主动构建场景

#### Slide 07: 演进的核心逻辑

**标题**：技术突破 → 界面革命 → 范式转移

**内容结构**（流程图）：

```
技术突破 → 上下文同化能力提升 → 界面革命 → 范式转移
    ↑                                              ↓
    └──────────────── 反馈循环 ──────────────────┘
```

**关键洞察**：

1. **更高智能 = 更低交互成本**

   - 机器越智能，Context Engineering 越自然
   - 人类投入的"熵减努力"越少

2. **质变而非量变**：

   - 每个阶段是范式革命，而非渐进改良
   - GUI、NLP、多模态各代表一次跃迁

3. **硅基智能加速**：
   - 碳基智能（人类）进化缓慢
   - 硅基智能（机器）迭代快速
   - 差距缩小是 Context Engineering 进步的根本驱动

**设计要点**：

- 循环流程图
- 关键词高亮
- 使用箭头强调因果关系

#### Slide 13: 上下文收集与存储

**标题**：上下文收集：从单一设备到分布式生态

**内容结构**：

**两大设计原则**：

1. **最小充分原则**：只收集任务所需的必要信息
   - 价值在于充分性，而非数量
2. **语义连续性原则**：维持意义的连续，而非数据的连续

**收集维度的演进**：

**Era 1.0**：

- 单设备（PC、早期智能手机）
- 有限传感器（GPS、时钟、键鼠事件）
- 本地日志文件

**Era 2.0**：

- 多端点（手机、可穿戴、家庭传感器、云服务）
- 多模态信号流（文本、图像、音频、传感器）
- 分层存储架构

**存储策略（表格）**：
| 层级 | 存储位置 | 用途 | 示例 |
|------|----------|------|------|
| 短期 | 内存缓存、边缘节点 | 快速访问 | 活动记录 |
| 中期 | 本地数据库（SQLite, LevelDB） | 会话持久化 | 用户偏好 |
| 长期 | 云存储、远程数据库 | 跨设备同步 | 历史对话 |

**长时任务的挑战**：

- Context Window 短期且有限
- 需要外部持久化内存
- 示例：Claude Code 的结构化笔记、Pokemon 游戏代理的千步策略

**设计要点**：

- 清晰的表格展示
- 使用层级结构
- 底部案例说明

### **第五部分：设计考量 II - 管理（3 页）**

#### Slide 14: 文本上下文处理策略

**标题**：文本上下文管理：从原始到结构化

**内容结构**（对比卡片）：

**策略 1：时间戳标记**

- ✅ 优点：简单、低维护成本
- ❌ 缺点：无语义结构、线性增长、难检索
- 适用：简单聊天机器人

**策略 2：功能标签分类**

- ✅ 优点：语义清晰、易检索
- ❌ 缺点：略显刚性、限制灵活推理
- 适用：任务型对话系统

**策略 3：QA 对压缩**

- ✅ 优点：检索效率高
- ❌ 缺点：破坏思路连贯性
- 适用：FAQ、搜索引擎

**策略 4：层次化笔记**

- ✅ 优点：清晰结构、易浏览
- ❌ 缺点：难表达因果关系、时间演化
- 适用：Claude Code、SII CLI

**设计要点**：

- 2×2 卡片布局
- 每个策略：标题 + 优缺点 + 适用场景
- 使用图标（✅❌）增强可读性

#### Slide 15: 多模态上下文融合

**标题**：多模态上下文：异构信息的统一表示

**核心挑战**：

- 模态异构性：文本（离散序列）、图像（高维空间）、音频（时序连续）
- 信息密度差异
- 时间动态不同

**三种融合策略**：

**策略 1：映射到共享向量空间**

- 各模态独立编码器 → 投影层 → 共享嵌入空间
- 语义相关内容距离接近
- 代表：CLIP, ALIGN

**策略 2：联合自注意力**

- 投影后的 Token 统一送入 Transformer
- 文本和视觉 Token 相互注意
- 代表：GPT-4V, Claude 3

**策略 3：跨模态交叉注意力**

- 一个模态作为 Query，另一个作为 Key/Value
- 灵活的信息检索
- 代表：Qwen2-VL, Flamingo

**视觉元素**：

- 三个并列的架构示意图
- 使用箭头表示信息流
- 颜色区分不同模态

**设计要点**：

- 三栏布局
- 每栏：标题 + 示意图 + 代表系统
- 底部强调"人脑无需固定映射，AI 尚需设计"

#### Slide 16: 上下文组织：分层内存 + 隔离

**标题**：上下文组织：分层架构与功能隔离

**上半部分：分层内存架构**

**类比**：LLM = CPU, Context Window = RAM

**两层模型**：

1. **短期内存 (Short-term)**：

   - 定义：高时间相关性的上下文子集
   - 公式：`M_s = f_short(c ∈ C : w_temporal(c) > θ_s)`
   - 特点：快速检索、易失性

2. **长期内存 (Long-term)**：

   - 定义：高重要性的处理和抽象子集
   - 公式：`M_l = f_long(c ∈ C : w_importance(c) > θ_l)`
   - 特点：持久化、跨会话

3. **转移机制**：
   - `f_transfer: M_s → M_l`
   - 触发因素：重复频率、情感重要性、知识关联

**下半部分：上下文隔离**

**Subagent 模式**（Claude Code）：

- 每个 Subagent 独立上下文窗口
- 自定义系统提示 + 受限工具权限
- 避免主上下文污染

**轻量化引用**：

- 大体量数据外部存储（Sandbox）
- 窗口中仅保留引用
- 按需检索完整数据

**设计要点**：

- 上下分栏布局
- 公式居中展示
- 使用流程图展示转移机制

### **第六部分：设计考量 III - 使用（4 页）**

#### Slide 17: 上下文共享：系统内 vs 跨系统

**标题**：上下文共享：多智能体协作的基础

**上半部分：系统内共享（Intra-System）**

**模式 1：嵌入式提示**

- 前一智能体上下文直接放入下一个提示
- 示例：AutoGPT, ChatDev
- 优点：简单直接
- 缺点：提示膨胀

**模式 2：结构化消息**

- 固定 Schema：任务类型、输入、输出、推理步骤
- 示例：Letta, MemOS
- 优点：清晰一致
- 缺点：需要协议设计

**模式 3：共享内存**

- 间接通信：读写公共内存空间
- 变体：Memory Blocks (MemGPT) / Blackboard (分段) / Graph (TME, G-Memory)
- 优点：异步协作
- 缺点：需要协调机制

**下半部分：跨系统共享（Cross-System）**

**挑战**：不同系统 = 不同格式 + 不同逻辑

**策略 1：适配器转换**

- 各系统保持自有格式 + 转换器
- 优点：灵活性高
- 缺点：需为每对系统构建适配器

**策略 2：共享表示**

- 统一格式（JSON Schema / API）
- 自然语言摘要
- 语义向量
- 优点：避免多次转换
- 缺点：需要前期协商

**设计要点**：

- 上下分栏
- 使用流程图展示共享模式
- 表格对比不同策略

#### Slide 18: 上下文选择：相关性过滤

**标题**：上下文选择：注意力之前的注意力

**核心问题**：

> "并非所有可用上下文都有用。如何选择最相关的子集？"

**经验观察**：

- AI 编码性能在 Context Window > 50% 时下降
- 过度和不足都有害

**选择维度**：

**1. 语义相关性**

- 向量相似度检索（FAISS 等）
- 最近邻搜索
- 应用：RAG 管道

**2. 逻辑依赖性**

- 任务依赖前序步骤输出
- 构建依赖图（MEM1）
- 应用：推理链追踪

**3. 时效性与频率**

- 最近使用 + 访问频率
- 启发式优先级调整
- 应用：缓存策略

**4. 去重与合并**

- 检测语义重叠
- 保留最新/最详细版本
- 应用：主动内存管理

**5. 用户偏好与反馈**

- 学习用户习惯
- 调整记忆条目权重
- 应用：自进化内存

**过滤策略示例**（表格）：
| 策略 | 分块方式 | 检索方式 | 代表系统 |
|------|----------|----------|----------|
| 语义检索 | 固定窗口/AST | 向量相似度 | 通用 RAG |
| 非语义检索 | 行/Token | Grep/Regex | CLI 工具 |
| 结构化检索 | 函数/类/模块 | 知识图谱 | Windsurf |
| 重排序 | - | LLM 二次评分 | 高级 RAG |

**设计要点**：

- 清晰的维度列表
- 表格对比不同策略
- 使用图标增强可读性

#### Slide 19: 主动推断 + 终身保存

**标题**：主动推断与终身上下文的挑战

**上半部分：主动用户需求推断**

**从被动响应到主动预测**：

**策略 1：学习用户偏好**

- 分析对话历史 + 个人文档
- 识别沟通风格、兴趣模式
- 构建动态用户画像

**策略 2：推断隐藏目标**

- 分析查询序列（如：Python 装饰器 → 性能调优 → 软件设计）
- 多步逻辑推导（Chain-of-Thought）
- 预测潜在需求

**策略 3：主动提供帮助**

- 检测用户挣扎（犹豫、多次尝试）
- 主动提供工具（可视化、检查清单）
- 改善决策质量

**下半部分：终身上下文保存的四大挑战**

**挑战 1：存储瓶颈**

- 如何在资源受限下保留所有相关上下文？
- 需要：高压缩 + 高精度检索 + 低延迟访问

**挑战 2：处理性能下降**

- Transformer 的 O(n²) 复杂度
- 注意力稀释 + 长距离依赖困难
- 检索噪音增加

**挑战 3：系统不稳定性**

- 记忆累积导致小错误扩散
- 缺乏边界和验证机制
- 安全性隐患

**挑战 4：评估困难**

- 现有基准仅测检索能力
- 缺乏矛盾检测、撤销机制、推理链追溯
- 难以信任和改进

**未来方向**：

> **语义操作系统 (Semantic OS)**：像人类大脑一样成长的记忆系统
>
> - 大规模语义存储
> - 类人记忆管理（添加、修改、遗忘）
> - 超越 Transformer 的长程推理架构
> - 可解释的推理链追溯

**设计要点**：

- 上下分栏
- 挑战部分用警示色标注
- 未来方向用高亮框强调

#### Slide 21: 挑战与未来方向

**标题**：迈向 Context Engineering 3.0/4.0

**当前挑战**：

**1. 收集效率低下**

- 仍依赖显式用户输入
- 用户难以清晰表达意图
- 方向：更自然的多模态收集 + 意图推断

- 上下文快速积累
- 组织和检索困难
- 方向：可扩展的分层存储架构

- 复杂逻辑推理困难
- 图像关系信息处理弱
- 方向：增强语义推理和多模态对齐

**4. 长上下文性能瓶颈**

- Transformer O(n²) 复杂度
- 替代方案（Mamba, LongMamba）仍不完善
- 方向：新型架构 + 选择性注意力机制

**5. 上下文选择精度**

- 有用信号遗漏
- 噪音和冗余持续存在
- 方向：更精准的自适应选择策略

  - 像人类一样"遗忘"和"回忆"

  - "上帝视角"理解人类意图
  - 主客体关系反转
  - 发现人类未显式表达的需求
  - 示例：AlphaGo 教职业棋手新策略
